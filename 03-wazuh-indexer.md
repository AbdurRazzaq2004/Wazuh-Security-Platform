# 🗄️ Wazuh Indexer — The Database That Stores Everything

## What is the Wazuh Indexer?

The **Wazuh Indexer** is a highly scalable, full-text **search and analytics engine** that stores all the security alerts and events generated by the Wazuh Server.

> **In simple words:** If the Wazuh Server is the detective that finds problems, the **Wazuh Indexer is the filing cabinet** where all the evidence is stored, organized, and made searchable. Need to find all failed logins from last Tuesday? The Indexer finds them in milliseconds.

---

## 🎯 What Does It Do?

| Function | Description |
|---|---|
| **Indexes data** | Organizes security events for fast searching |
| **Stores alerts** | Keeps all security alerts in a structured format |
| **Search engine** | Allows near real-time searching across millions of events |
| **Analytics** | Powers data analytics and visualizations on the Dashboard |
| **Long-term storage** | Retains historical data for compliance and forensics |
| **High availability** | Supports clustering for redundancy and scalability |

---

## 🏗️ How Does It Work?

### Data Format: JSON Documents
The Wazuh Indexer stores everything as **JSON documents**. Each alert or event is a JSON document with fields like:

```json
{
  "agent": {
    "name": "web-server-01",
    "ip": "10.0.1.50",
    "id": "003"
  },
  "rule": {
    "description": "SSH brute-force attack detected",
    "level": 10,
    "id": "5712"
  },
  "data": {
    "srcip": "192.168.1.100",
    "dstuser": "root"
  },
  "timestamp": "2026-02-19T10:30:15.000Z"
}
```

Each document has **keys** (field names) and **values** (strings, numbers, booleans, dates, arrays, etc.)

---

### Indices: Collections of Documents
An **index** is a collection of related documents — like a folder full of similar files.

> **Simple Analogy:** Think of a library. An **index** is like a bookshelf for a specific topic. `wazuh-alerts-*` is the shelf for all security alerts. `wazuh-archives-*` is the shelf for all raw log archives.

**Common Wazuh Indices:**

| Index Pattern | What It Stores |
|---|---|
| `wazuh-alerts-*` | Security alerts generated by rules |
| `wazuh-archives-*` | All collected logs (including non-alerts) |
| `wazuh-monitoring-*` | Agent monitoring status data |
| `wazuh-statistics-*` | Wazuh performance statistics |

---

### Shards: Distributed Storage
Documents within an index are split into **shards** — smaller pieces distributed across multiple nodes.

> **Simple Analogy:** Instead of putting all your eggs in one basket, you split them across multiple baskets. If one basket breaks, you still have the others. That's what shards do.

```
                    Index: wazuh-alerts-2026.02
                    ┌──────────────────────────┐
                    │                          │
        ┌───────────┼───────────┬──────────────┘
        ▼           ▼           ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐
   │ Shard 0 │ │ Shard 1 │ │ Shard 2 │
   │ Node 1  │ │ Node 2  │ │ Node 3  │
   └─────────┘ └─────────┘ └─────────┘
        │           │           │
        ▼           ▼           ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐
   │Replica 0│ │Replica 1│ │Replica 2│
   │ Node 2  │ │ Node 3  │ │ Node 1  │
   └─────────┘ └─────────┘ └─────────┘
```

**Why shards matter:**
- **Redundancy:** If a node (machine) fails, replicas on other nodes keep your data safe
- **Performance:** Queries can be processed in parallel across multiple shards
- **Scalability:** Add more nodes to handle more data

## ⚡ Near Real-Time Search

One of the killer features: the Indexer provides **near real-time** search.

> **What this means:** When the Wazuh Server sends an alert at 10:30:15, it's typically **searchable within 1 second** on the Indexer. So when you open the Dashboard, you're seeing events that happened just a moment ago.

| Feature | Details |
|---|---|
| **Indexing latency** | ~1 second from ingestion to searchable |
| **Search speed** | Milliseconds across millions of documents |
| **Full-text search** | Search any field, any value, any combination |
| **Aggregations** | Calculate stats, counts, averages in real-time |

## 🏢 Deployment Options

### Single-Node (Simple)
```
┌──────────────────────┐
│    Wazuh Indexer      │
│   (One Machine)       │
│                       │
│  All shards here     │
└──────────────────────┘
```
- Good for **small environments**
- Simple to set up and manage
- **No redundancy** — if the node goes down, search is unavailable

### Multi-Node Cluster (Production)
```
┌──────────────────────────────────────────────────┐
│               Wazuh Indexer Cluster                │
│                                                    │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐  │
│  │  Node 1    │  │  Node 2    │  │  Node 3    │  │
│  │ (Primary   │  │ (Replica   │  │ (Replica   │  │
│  │  shards)   │  │  shards)   │  │  shards)   │  │
│  └────────────┘  └────────────┘  └────────────┘  │
│                                                    │
└──────────────────────────────────────────────────┘
```
- **High availability:** One node fails? Others keep serving
- **Scalable:** Add more nodes for more capacity
- **Load balanced:** Queries distributed across nodes

## 🧰 Built-in Features

The Wazuh Indexer comes with several powerful features out of the box:

| Feature | What It Does | Why You Care |
|---|---|---|
| **Data Roll-ups** | Compresses old data into summaries | Save storage space |
| **Alerting** | Trigger actions based on indexed data patterns | Automated notifications |
| **Anomaly Detection** | ML-based detection of unusual patterns | Finds unknown threats |
| **Index Lifecycle Management (ILM)** | Automatically manages index aging (hot → warm → cold → delete) | Optimize storage costs |
| **Snapshots** | Backup indices to external storage | Disaster recovery |
| **Security** | TLS encryption, RBAC, audit logging | Enterprise security |

## 📊 What Data Does It Store?

The Indexer stores **four main types** of data:

### 1. Security Alerts 🚨
Events that matched a detection rule on the Wazuh Server.
- Brute-force attempts
- Malware detections
- Vulnerability findings
- Configuration violations

### 2. Raw Data Events 📝
All events sent by agents, including those that didn't trigger alerts (when archives are enabled).
- System logs
- Application logs
- Authentication events

### 3. Inventory Data 📦
System inventory collected by agents.
- Installed software and versions
- Open ports
- Running processes
- Network interfaces

### 4. Agent Monitoring Data 📡
Status and health information about agents.
- Agent connection status
- Last keep-alive time
- Agent version and OS

## 🔌 How Data Gets In

```
Wazuh Agent ──▶ Wazuh Server ──▶ Filebeat ──TLS──▶ Wazuh Indexer (Port 9200)
                (Analysis)       (Shipping)          (Storage & Search)
```

1. **Agents** collect data and send to the **Server**
2. **Server** analyzes data and generates alerts
3. **Filebeat** reads the Server's output
4. **Filebeat** sends data to the **Indexer** over **TLS** on port **9200**
5. **Indexer** indexes and stores the data
6. **Dashboard** queries the Indexer to show you the results
